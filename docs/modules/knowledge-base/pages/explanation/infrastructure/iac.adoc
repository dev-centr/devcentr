= Infrastructure as Code (IaC)

Infrastructure as Code means you store your server, network, and cloud setup in files (code) instead of clicking in a web console. Those files can be versioned, reviewed, and repeated so your infra is consistent and auditable.

== Why OpenTofu by default

DevCentr recommends **OpenTofu** as the default IaC tool when you add or install IaC support.

* OpenTofu is **open source**. You are not tied to a single vendor.
* It is **compatible** with Terraform: it reads the same HCL configs and state, so existing Terraform users can switch without rewriting everything.
* It is maintained by the Linux Foundation and a broad vendor-neutral community.

You can still use Terraform, Pulumi, or other tools; the app supports multiple IaC styles. OpenTofu is the suggested starting point for new setups.

== Personal use: repo-level IaC

For **personal repos** (your own account, not an organization), IaC is usually **repo-level**: you put the infra code **inside the same repo** as the project that uses it.

* One repo, one project. An `infra/` folder (or similar) in that repo holds your OpenTofu files (e.g. `main.tofu`, `variables.tofu`, `backend.tofu`).
* You own both the app and the infra; there is no separate “central infra repo” to coordinate with.
* This is the **default suggestion** when you add IaC from DevCentr while browsing one of your personal repositories: add IaC to this repo, recommend OpenTofu, bootstrap a minimal layout.

If you have **many personal projects** and want one place to manage shared infra (e.g. a single “my-infra” repo), you can do that too; DevCentr can offer that as an optional pattern. For most personal use, repo-level IaC per project is enough and keeps things simple.

== Organization use: different options

When you work in an **organization** (company or team org), the picture is different. The org may want a single place for shared infra, or allow per-team IaC with some governance. The app and docs adapt to that.

=== Org-level: one infrastructure repo

Many organizations use a **single repo** (e.g. `infrastructure` or `platform-infra`) for the whole org.

* All shared infra (networks, databases, IAM, environments) lives there.
* Application repos usually contain **no** IaC, or only thin wrappers that reference the central repo.
* Pros: one place to review and secure infra, no duplicated state across many repos, easier to enforce standards.
* Best when: the org has multiple services and wants consistency and a platform or infra team.

When you browse an **org** in DevCentr, the installation workflow can offer to set up or manage this kind of org-level infrastructure repo.

=== Per-project IaC in an org

Teams can still keep IaC **inside their app repo** (e.g. `infra/` per repo). The org may then want visibility or guardrails without taking the code into a central repo. That leads to “delegation” patterns below.

=== Hybrid (org)

A common pattern is **both**: a central infra repo for shared building blocks (VPCs, DB clusters, IAM), and **thin** IaC in app repos that only deploy app-specific resources and consume outputs from the central repo.

When you use the DevCentr flow in an **org** context, you can choose:

* Set up or manage an **org-level** infrastructure repo (if the org has or wants shared infra).
* Add **per-project** IaC to a repo (with the decision flow and, if relevant, orphaned-IaC and registration options below).

== Orphaned IaC and registering with the central repo (org only)

This section applies only when you are in an **organization** that already has a central infrastructure repo.

If a **project repo** in that org also contains its own IaC, that project IaC is sometimes called **orphaned**: it lives outside the org’s main infra model and can be hard to govern or discover.

DevCentr can **detect and mark** orphaned IaC (e.g. when an org-level infra repo exists and a project repo has its own IaC). It can then **assist you in registering** that project’s infra with the central repo using one of the patterns below, so the org has visibility and guardrails without forcing every team to move all their code into one repo.

For **personal use**, there is no “central org repo,” so orphaned IaC and registration do not apply unless you have chosen to maintain a personal infra repo and want to register a project to it.

== Delegating and distributing IaC in large teams (org only)

This section is for **organizations**. Teams want **autonomy**; the platform org wants **consistency and visibility**. You can give teams control over their IaC without putting every detail in a central repo. The central repo acts as a **registry and orchestrator**, not the only place IaC lives.

Below are patterns that work in practice.

=== Pattern 1: Registration via metadata (most scalable)

Teams keep their IaC in their **own repo** (e.g. `my-service/infra/main.tofu`). The **central infra repo** only holds a **registration file** per service (e.g. `services/my-service.yaml`) with:

* Repo URL
* Environment names
* Required modules
* Outputs the service exposes to the org
* Compliance or ownership metadata

The platform pipeline reads these registrations and validates the team’s IaC against org policies, allowed module versions, and state backend rules. The central repo **never sees** the team’s actual `.tofu` files. Used by many platform-engineering orgs and “service catalog” setups.

=== Pattern 2: Thin wrapper modules

Teams write full IaC internally but expose only a **thin module** to the org (e.g. a small public interface under `infra/modules/public/service_interface/`). The central repo imports only that module (e.g. via a `source` pointing at that path). The org sees only the public interface; teams can change internals freely. Similar in spirit to AWS CDK constructs or Kubernetes operators exposing a public API.

=== Pattern 3: Remote state linking only

Teams keep full IaC in their repos. The central repo only **references remote state outputs** (e.g. `terraform_remote_state` or OpenTofu equivalent) to read outputs and state location. The central repo sees no internal resources, only what teams expose. Good when teams are mature and the org wants minimal coupling.

=== Pattern 4: Policy as code, decentralized IaC

Teams keep IaC in their repos. The central repo stores **policies** (e.g. OPA/Conftest), module version constraints, security rules, and naming conventions. CI runs these policies against team repos; the central repo never touches team IaC directly. Common with Open Policy Agent, Terraform Cloud/Enterprise, Atlantis, or GitHub Actions with policy gates.

When you use DevCentr to register project IaC with the central repo, the flow can guide you toward one of these patterns (e.g. “add a registration file” vs “expose a thin module”) depending on what your org supports.
