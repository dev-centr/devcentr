= Git GUIs and Visual History Browsers

Git GUIs aim to make version control easier, but most of them share the same limitations:

* They focus on **lists** (commits, files, hunks) rather than high-level **visuals**.
* They rarely provide a **persistent file tree** that you can scrub through history.
* They make it hard to **see the shape and volume** of work over time.

== Common limitations

* *No persistent tree:* Many clients only show a list of changed files for the selected commit. The tree view is either missing or rebuilt per-commit, making it difficult to see how entire folders evolve (adds/removes/renames).
* *Commit history as a flat list:* The main log is usually a simple vertical list. This hides:
** The relative size of commits.
** Patterns over time (bursts of activity, quiet periods).
** How different areas of the repo are affected over time.
* *Weak visual diffing of folder hierarchy:* Few tools let you "rewind" the file hierarchy and visually compare folders between points in time.

== DevCentr Git viewer goals

DevCentr's built-in Git viewer is designed as a **project browser with Git controls**, not just a commit list:

* A **persistent project tree** that remains visible while you scrub through history.
* Folder and file nodes visually indicate:
** Adds and removes over time.
** Renames and moves.
** Hot spots with frequent changes.
* A **timeline-style commit view** that makes the weight of each commit obvious at a glance.

=== Horizontal timeline with scrubber

The viewer models commit history as a horizontal timeline:

* HEAD is on the **right**, selected by default.
* Each commit is rendered as a **block**:
** Width and/or color intensity can reflect size (lines changed, files touched).
** Commits with more impact stand out visually.
* A **timeline scrubber** (like a film editor playhead) lets you:
** Drag left/right to move through history.
** Zoom in and out on time ranges for detail or overview.

This gives an immediate sense of:

* Where big refactors or migrations happened.
* Which time periods were most active.
* How recent your currently checked-out state is relative to the overall history.

=== Replay index for fast seeking

Naively, a Git viewer needs to ask Git to recompute state for each commit. For long histories, this can be too slow for "instant" seeking.

DevCentr's Git viewer addresses this with a **replay index**:

* An indexer process walks the commit history and records:
** Metadata for each commit (lines added/removed, files changed).
** Enough information to reconstruct tree snapshots quickly.
* The timeline marks commits as:
** *Indexed:* Normal appearance; seeking is fast and serviced from the index.
** *Indexing in progress:* Commit boxes outlined in purple while the indexer is working through them.
** *Unindexed:* Greyed out for commits beyond the current index frontier.

Indexing behavior:

* For **normal-sized repositories**, indexing runs automatically by default.
* For repositories with **very large histories**, automatic indexing is disabled:
** The viewer shows a *Set up quick seek* button.
** Clicking it opens a dialog that:
*** Explains the indexing process.
*** Shows the total number of commits.
*** Warns about potential runtime.
*** Offers a *Start* button.
* Indexing can be:
** **Paused** by the user (stop button acts as pause).
** **Resumed** later from where it left off.

While the indexer is running:

* Seek requests inside the currently indexed range are handled via the indexer for quick responses.
* Seeks beyond the indexed range fall back to Git's normal mechanisms or are clearly marked as slower operations.
